# Project Governance

## Principles

The OOXML Behavioral Intelligence Project is:

- **Vendor-neutral** — no single company or product controls the project direction
- **Open-source** — all code and documentation are freely available under open licenses
- **Non-commercial core** — the knowledge base, tools, and datasets remain free and open
- **Community-driven** — decisions are made through open consensus-building

## Project Structure

### Maintainers
Maintainers are trusted contributors who have demonstrated sustained commitment. They:
- Review and merge pull requests
- Manage the issue tracker
- Steward the project roadmap

### Contributors
Anyone who submits a pull request, opens a well-researched issue, or contributes documentation or datasets.

### Technical Steering Committee (TSC)
As the project grows, a TSC will be formed with representation from:
- Open-source office suite projects
- Academic researchers
- Independent contributors
- Industry partners

## Decision Making

1. **Lazy consensus** — proposals with no objections within 14 days are considered approved
2. **Contentious decisions** — escalated to maintainers for vote (simple majority)
3. **Architectural decisions** — require at least 2 maintainer approvals

## Roles & Responsibilities

| Role | Access | Responsibility |
|------|--------|---------------|
| Maintainer | Write access | PR review, roadmap stewardship |
| Contributor | Fork-based | Code, docs, datasets |
| Community | Issues/Discussions | Feedback, bug reports, ideas |

## Becoming a Maintainer

Contributors who demonstrate:
- Consistent quality contributions over 6+ months
- Constructive code reviews and community participation
- Alignment with project values

...may be nominated by existing maintainers and approved by consensus.

## Code of Conduct Enforcement

Violations of the code of conduct will be handled by maintainers. Sanctions may include:
- Warning
- Temporary ban
- Permanent ban

## Long-Term Governance Strategy

This project is designed to outgrow its founder. The governance model evolves through three phases:

### Phase 1 — Founder-Led (Current)
- Single maintainer (project founder)
- Direct community input via Issues and Discussions
- Rapid iteration, low process overhead

### Phase 2 — Steering Committee (12-24 months)
- Multi-stakeholder TSC formed
- Representation from contributing organizations
- Formal RFC process for major changes
- Maintainer elections

### Phase 3 — Foundation Transfer (24-48 months)
Transfer to an independent open-source foundation. Evaluated hosts:

| Foundation | Strengths | Considerations |
|-----------|-----------|----------------|
| **The Document Foundation** | Natural alignment with LibreOffice ecosystem | May create perceived vendor dependency |
| **Apache Software Foundation** | Strong governance, vendor-neutral reputation | Process overhead, incubation requirements |
| **Linux Foundation** | Broad industry reach, project maturity support | May not specialize in document formats |
| **Sovereign Tech Fund** | European public-interest funding | Narrower scope, application-based |

**Evaluation criteria for foundation selection:**
1. Vendor neutrality and independence
2. Ability to attract diverse stakeholders
3. Infrastructure and legal support
4. Alignment with project values
5. Community preference (RFC process)

### Transfer Conditions

A foundation transfer is appropriate when:
- The project has 5+ active maintainers from different organizations
- The knowledge base exceeds 100,000 compatibility cases
- Multiple organizations depend on the project's outputs
- The founder's continued involvement is not a bottleneck

## Funding & Sponsorship

The project may accept sponsorships and grants, provided they:
- Do not create conflicts of interest
- Are transparently disclosed
- Do not grant control over project direction
- Do not restrict open access to data or tools

## Changes to Governance

This document may be amended by maintainer consensus with a 30-day comment period. Foundation transfer requires TSC vote and community ratification.
